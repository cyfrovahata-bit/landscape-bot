# Специфікація: Дорожній табель (Telegram Mini App)

Документ описує продукт повністю і остаточно. Він самодостатній: розробнику не потрібно звертатись до попередніх версій специфікації, аудитів чи історії змін, щоб реалізувати або підтримувати цю частину системи.

---

## 1. Призначення

Дорожній табель — інструмент бригадира ландшафтної/будівельної бригади для обліку робочого дня: авто, люди, обʼєкти, виконані роботи, обсяги, пробіг. Використовується щодня по 8–12 годин, здебільшого однією рукою, часто за кермом або в русі. Google Таблиці лишаються основним джерелом істини для довідників (авто, люди, обʼєкти, роботи); мінідодаток пише туди ж, а Postgres — швидкий кеш для читання.

Головний принцип: **бригадир ніколи не втрачає доступ до дня**. Дані автозберігаються локально, звіт видно й редаговано аж до моменту, коли адміністратор його затвердить.

---

## 2. Архітектурна концепція

Застосунок побудований навколо **HUB** — центрального екрана поїздки, а не лінійного візарда. HUB відкриває підменю (Авто, Люди, Обʼєкти, Роботи), кожне з яких повертає назад у HUB. Будь-який параметр дня редагується в будь-який момент — до виїзду, під час руху, на обʼєкті, навіть після відправки звіту (доки він не затверджений).

Другий принцип: **об'єкт — не етап, а стан**. Немає жорсткого переходу "обʼєкт А завершено → обʼєкт Б розпочато". Бригадир може відвідати обʼєкт, поїхати далі, повернутись, розпочати роботи вдруге. Порядок відвідування — довідковий, не примусовий.

Третій принцип: **людина має власний, окремий облік часу**. Час і оплата рахуються по кожному працівнику окремо, а не по групі загалом — часткові виїзди частини бригади не спотворюють нарахування.

## 3. Модель даних

### 3.1 Довідники (Google Sheets → Postgres, read-only з мінідодатка)

```ts
type Car      = { id: string; name: string; plate: string | null; active: boolean };
type Employee = { id: string; name: string; brigadeId: string | null; position: string | null; active: boolean };
type Brigade  = { id: string; name: string };                         // назва бригади — окреме поле, НЕ виводиться з посади лідера
type Work     = { id: string; name: string; category: string | null; unit: string | null; tariff: number; active: boolean };
type WorkObject = { id: string; name: string; address: string | null; city: string | null; active: boolean };  // city — окреме нормалізоване поле, не парситься з address
```

`Brigade.name` і `WorkObject.city` — самостійні поля довідника (синхронізуються з відповідних колонок джерела). Групування людей по бригадах і обʼєктів по містах на екранах вибору спирається на ці поля, а не на текстовий парсинг посади чи адреси — це критично при десятках бригад і сотнях обʼєктів, де евристичний парсинг неминуче почне давати збої.

### 3.2 Стан поточного дня (клієнтський стан однієї поїздки)

```ts
type PresenceEntry = {
  employeeId: string;
  from: string;     // ISO — коли людину висадили на обʼєкті / коли почалась робота
  to?: string;       // ISO — коли забрали / відправили додому / завершили роботи. Відсутнє = ще триває
};

type PlannedWork = { workId: string; workName: string; unit: string; volume: string }; // volume: "" не введено, "?" свідомо відкладено

type ObjPlan = {
  objectId: string;
  objectName: string;
  works: PlannedWork[];
  here: string[];                 // хто фізично висаджений на обʼєкті ЗАРАЗ (миттєвий стан, не історія)
  presence: PresenceEntry[];      // журнал відпрацьованого часу: інтервали, коли людина була В РОБОТІ (не просто фізично тут)
  visited: boolean;                // чи відвідано хоч раз (індикатор прогресу маршруту, НЕ блокує повторний вхід)
  notes: string;
  photoUrls: string[];
};

type EmployeeDayState = {
  employeeId: string;
  sentHome: boolean;               // явний стан "поїхав додому" — термінальний на сьогодні
  sentHomeAt?: string;
};

type CoefPair = { disciplineCoef: number; productivityCoef: number };

type DayState = {
  date: string;
  carId: string;
  odoStart: string;
  odoStartPhoto: string | null;
  odoEnd: string;
  odoEndPhoto: string | null;
  employeeIds: string[];           // хто в бригаді сьогодні (роз'їзний склад)
  employeeStates: Record<string, EmployeeDayState>;
  onboard: string[];                // хто зараз у машині
  plans: ObjPlan[];
  coefs: Record<string, CoefPair>;  // денні коефіцієнти дисципліни/продуктивності на робітника
  tripStartedAt: string | null;
  step: Step;
};
```

**Presence замінює поняття "зміни на обʼєкті".** Одна дія "▶️ Почати роботи" відкриває `PresenceEntry` (`from = now`) для кожного, хто зараз `here` і ще не має відкритого запису на цьому обʼєкті. Це автоматично підтримує:
- **повторний вхід на обʼєкт** — новий виклик "Почати роботи" пізніше за день просто додає нові записи, старі закриті записи лишаються історією;
- **приєднання людей пізніше** — якщо частину бригади довисадили, коли робота вже йде, повторне натискання "Почати роботи" відкриє запис лише для новоприбулих, не займаючи вже працюючих;
- **частковий виїзд** — забрати конкретну людину (`pickUpHere` на неї саму) закриває **лише її** запис (`to = now`), решта продовжують працювати з незмінним `from`.

Час людини за день = сума `(to ?? now) − from` по всіх її записах `presence` на всіх обʼєктах. Це і є коректний, по-людський облік часу, що прямо йде в розрахунок оплати.

**`sentHome` — окремий явний стан**, не видалення з поїздки. Людина лишається в `employeeIds` (усі вже нараховані години зберігаються), але:
- зникає з пулів "доступний тут / у машині" для нових призначень;
- позначається на HUB як "🏠 поїхав додому";
- при відправці додому автоматично закриваються всі її відкриті `presence`-записи.

Видалення людини з `employeeIds` (а не `sentHome`) допустиме лише **до виїзду** (`tripStartedAt === null`) — це виправлення помилки планування, а не робоча подія дня. Після виїзду єдиний спосіб прибрати людину з активної роботи — `sentHome`.

`here` і `presence` навмисно не зведені в одну сутність: `here` — це "де людина фізично зараз" (потрібно для UI висадки/підбору, миттєве, без історії), `presence` — це "скільки часу вона реально в роботі" (потрібно для оплати, накопичувальне). Людина може бути в `here`, ще не маючи відкритого запису в `presence` (висаджена, але роботу ще не почали) — це не помилка, а нормальний проміжний стан.

## 4. State Machine

### 4.1 Рівень дня

```
PLANNING ──(Виїхати, з підтвердженням готовності)──► ON_ROUTE
ON_ROUTE ──(Приїхали на базу)──► RETURNED
RETURNED ──(Далі → Підсумок дня)──► REVIEW
REVIEW ──(Відправити, з Popup-підтвердженням)──► SUBMITTED
SUBMITTED ──(адміністратор затверджує — зовнішня дія)──► APPROVED
```

`PLANNING`, `ON_ROUTE`, `RETURNED`, `REVIEW`, `SUBMITTED` — усі редаговані. HUB доступний з будь-якого з них (кнопка "✏️ Редагувати" на DRIVE/AT_OBJECT, "До меню поїздки" на READY, "Редагувати ще" на екрані підтвердження). Єдиний нередагований стан — `APPROVED`.

Перехід `PLANNING → ON_ROUTE` дозволений лише коли: обрано авто й уведено одометр старту, обрано хоча б одну людину, обрано хоча б один обʼєкт і на кожному є хоч одна робота.

Перехід `ON_ROUTE → RETURNED` дозволений, коли всі люди зі складу дня — або `onboard`, або `sentHome` (тобто нікого не забуто на обʼєкті).

`SUBMITTED` — не кінцевий стан застосунку, а помітка "відправлено, очікує на розгляд". Повторна відправка з `SUBMITTED` дозволена й перезаписує ті самі рядки дня, додаючи новий запис у журнал подій.

### 4.2 Рівень обʼєкта (не FSM, а набір незалежних прапорців — реальність недостатньо лінійна для строгого стану)

```
visited: boolean        — чи хоч раз відвідано (для позначки на маршруті/дашборді)
here.length > 0         — чи є зараз фізично люди
presence має відкритий запис — чи триває зараз робота
```

Будь-яка комбінація цих трьох прапорців легітимна одночасно: обʼєкт може бути `visited=true`, `here=[]`, без активних записів (повністю оброблений, усіх забрано) — і за годину туди можна знову довезти людей, знову відкривши записи присутності. Немає стану "обʼєкт закрито назавжди".

### 4.3 Рівень людини (за день)

```
NOT_ON_TRIP → PLANNED → ON_BOARD ⇄ AT_OBJECT(objId) ⇄ WORKING(objId)
                                                              │
                                                    (у будь-який момент)
                                                              ▼
                                                         SENT_HOME (термінально на сьогодні)
```

`WORKING(objId)` — має відкритий `PresenceEntry` на цьому обʼєкті. Переходи `ON_BOARD ⇄ AT_OBJECT ⇄ WORKING` можливі багато разів за день у будь-якому напрямку для будь-якого обʼєкта. `SENT_HOME` досяжний з будь-якого стану й на сьогодні незворотній (виправлення помилкового "відправив додому" — через `Undo`-тост одразу після дії, не пізніше).

## 5. Навігація та екрани

Верхній рівень — не вкладки, а один живий екран (HUB) з підменю-шторками. Кожне підменю має чіткий вхід і повертає туди, звідки відкрито.

### 🏠 HUB — головний екран поїздки

Завжди перший екран при вході в "Дорожній табель". Показує:
- Дату й, **поки триває поїздка — живий таймер** (`now − tripStartedAt`), незалежно від того, на якому подекрані зараз перебуває бригадир — таймер на HUB не зникає, коли виходиш подивитись/змінити щось інше.
- До виїзду — індикатор готовності "N/4 готово" (авто+одометр, люди, обʼєкти, роботи).
- 4 картки переходу: 🚙 Авто, 👥 Люди, 📍 Обʼєкти, 🧱 Роботи — кожна зі своїм статусом.
- Під час поїздки — два живі блоки:
  - **"Хто де зараз"**: кожен рядок людини клікабельний, відкриває міні-меню дій — "🔄 Перенести на інший обʼєкт" / "🏠 Відправити додому". Статус рядка: 🚗 в дорозі / 🔧 працює на [обʼєкт] / 📍 на [обʼєкт] (не працює) / 🏠 поїхав додому.
  - **"Маршрут"**: кожен обʼєкт зі статусом (заплановано / 🔧 роботи тривають / тут є люди / оброблено), клікабельний — відкриває "На обʼєкті" для будь-якого обʼєкта незалежно від того, відвідувався він раніше чи ні.
- "🔁 Повторити вчорашній маршрут" — карткою-пропозицією, якщо день ще порожній.
- "🕓 Історія дня" — розгортається список значущих подій з часом.
- "🗑 Почати день заново" — до виїзду, з Popup-підтвердженням.
- Головна дія знизу: "Далі → Перевірка перед виїздом" (до виїзду) або "↩️ Повернутися до поїздки" (під час).

### 🚙 Авто → Одометр

Список авто, зайняті іншим бригадиром сьогодні позначені й недоступні. При виборі авто поле одометра **одразу підставляється останнім відомим значенням** для цього авто (не просто підказка поруч — саме значення в полі, яке бригадир підтверджує чи коригує). Зміна авто скидає одометр і фото. Фото — за бажанням.

### 👥 Люди

Список згорнутий по бригадах (`Brigade.name`, не текст із посади), з пошуком по імені. Кнопка "✅ Обрати всю бригаду" в кожній групі. Зайняті іншим бригадиром — заблоковані з позначкою його імені; **статус зайнятості автоматично оновлюється** (періодичний реполл, поки бригадир перебуває на цьому екрані), а не фіксується один раз при вході.

### 📍 Обʼєкти

Список згорнутий по містах (`WorkObject.city`), з пошуком, вибір у кількох містах одночасно.

### 🧱 Роботи

Список обраних обʼєктів; тап відкриває вибір робіт для конкретного обʼєкта: пошук, "📋 Застосувати ці роботи до всіх обʼєктів", нотатки, фото. Обсяги на цьому етапі не вводяться — лише після фактичного виконання.

### ✅ Перевірка перед виїздом (READY)

Підсумок: авто+км, люди, обʼєкти з роботами. Кнопка "🚗 Виїхати".

### 🚗 У дорозі (DRIVE)

Живий таймер, підказка щодо наступного невідвіданого обʼєкта (**довідково, не примусово**). Список усіх обʼєктів маршруту з прямими діями по кожному рядку, без переходу на інший екран:
- **"➕ Додати обʼєкт"** (компактний інлайн-пошук/пікер прямо тут) і **"🗑"** на кожному рядку — редагування маршруту без виходу з DRIVE в HUB.
- **"🔽 Висадити тут"** / **"🔼 Забрати"** — на будь-якому обʼєкті, у будь-якому порядку.
- **"📍 Відкрити обʼєкт"** — на будь-якому рядку, відвіданому чи ні, відкриває "На обʼєкті" саме для нього.

Немає єдиної глобальної кнопки "Прибув на обʼєкт", привʼязаної до одного "наступного" обʼєкта — кожен рядок сам є точкою входу.

### 📍 На обʼєкті (AT_OBJECT)

Відкривається для будь-якого обʼєкта маршруту, скільки завгодно разів за день.
- Якщо є відкриті `presence`-записи — картка "Роботи тривають" з живим таймером (від найранішого відкритого запису), списком робіт і людей у роботі, кнопкою "⏹ Завершити" (закриває всі відкриті записи цього обʼєкта → веде до введення обсягів).
- "▶️ Почати роботи" — одна дія, відкриває записи присутності для всіх, хто зараз `here` і ще не в роботі. Доступна незалежно від того, чи вже є активні записи (для новоприбулих).
- "🔄 Перенести людей на інший обʼєкт" (з "Обрати всіх"), "👥 Висадити людей тут" — завжди доступні.
- Блок виїзду з обʼєкта візуально розділяє людей на **"🚐 Приїхали бусом"** і **"🚶 Приїхали самі"**. Перших можна однією дією посадити назад у бус. Біля кожного, хто приїхав сам, є окрема дія **"Посадити в бус"**; усі, хто лишився в цій групі, однією дією **"Зняти з обʼєкта"** їдуть самостійно. Обидва варіанти закривають лише відкриті записи вибраних людей та не змінюють вже нараховані години.
- "➡️ Продовжити маршрут" — **завжди активна**, без умов. Активна робота на обʼєкті триває у фоні (записи присутності лишаються відкритими) незалежно від того, де фізично перебуває бригадир.

### 📊 Обсяги

Список робіт обʼєкта з позначкою заповнено/не заповнено; деталь — велика цифра, одиниця виміру, клавіатура, опція "заповнити пізніше". Повертає туди, звідки відкрито (з обʼєкта чи з екрана "Повернення" — відстежується явно, не жорстко захардкожено).

### 🏁 Повернення (RETURN)

Список відвіданих обʼєктів і обʼєктів, де є люди, що прибули самі: для прибулих бусом — групова дія "Посадити в бус"; для прибулих самостійно — окрема "Посадити в бус" біля кожної людини та групова "Зняти з обʼєкта" для решти; також доступна "🟡 Ввести обсяги" (якщо є незаповнені — підстраховка, якщо забули на місці). Перехід до підсумку заблокований, доки хтось фізично лишається на обʼєкті. Одометр на фініші **перевіряється, що не менший за старт**. Фото — за бажанням.

### 📋 Підсумок дня (REVIEW)

Пробіг і клас поїздки, обʼєкти з роботами й обсягами, години кожного працівника (по факту сумованих `presence`-інтервалів), розгорнутий блок коефіцієнтів дисципліни/продуктивності, прев'ю розподілу фонду. Кнопка відправки — **з Popup-підтвердженням** ("Відправити звіт? Далі можна редагувати, поки адміністратор не затвердить"), текст кнопки — "📤 Оновити звіт", якщо це вже не перша відправка сьогодні.

### ✅ Після відправки

Показує підсумок щойно відправленого дня (пробіг, доплата, фонд по обʼєктах). Дві дії: "✏️ Редагувати ще" (лишитись і продовжити правити) і "До меню" (вийти). Це не кінцевий/замкнений екран — звіт лишається доступним і редагованим при будь-якому наступному відкритті табеля тим самим днем, аж до затвердження.

### 🔒 День затверджено

Показується **лише** якщо адміністратор фактично затвердив день. Read-only зведення + "🔓 Запросити редагування" (надсилає запит, не розблоковує автоматично).

---

## 6. Сценарій: Планування дня

Бригадир відкриває табель → HUB. Обирає авто (одометр підставляється автоматично) → Зберегти → назад у HUB. Обирає людей (пошук, групи бригад, масовий вибір бригади) → Зберегти. Обирає обʼєкти (групи міст, кілька міст одночасно) → Зберегти. Заходить у "Роботи" → для кожного обʼєкта обирає роботи (з опцією поширити пакет на всі обʼєкти) → Готово. Якщо це не перший день бригади — на HUB одразу пропонується повторити вчорашній маршрут одним тапом.

Будь-який із цих чотирьох кроків можна пройти в будь-якому порядку, повторити, змінити — HUB не має "правильної" послідовності.

---

## 7. Сценарій: Люди

Робочий склад дня (`employeeIds`) фіксується під час планування й лишається базою для розрахунку доплати за виїзд (ділиться порівну на всіх, хто їхав, незалежно від того, чи довелось комусь поїхати раніше). Після виїзду склад **не змінюється видаленням** — лише станом:

- **Поїхав додому.** Тап по людині в блоці "Хто де зараз" → "🏠 Відправити додому". Закриває її відкриті `presence`-записи в моменті дії (вона отримує оплату за фактично відпрацьований час, не більше й не менше), прибирає її з пулів "доступний" для нових призначень, позначає статус на дашборді. Дію можна скасувати одразу через Undo-тост.
- **Перенесено на інший обʼєкт.** Закриває `presence`-запис на старому обʼєкті (якщо був відкритий) для конкретно цієї людини, не чіпаючи інших там; на новому обʼєкті вона зʼявляється в `here`, роботу там потрібно почати окремою дією "Почати роботи" (або вона автоматично долучиться до вже відкритих записів цього обʼєкта тим самим механізмом "почати роботи для новоприбулих").
- **Частковий виїзд бригади з обʼєкта.** Забір частини людей (з обʼєкта чи через швидку дію на DRIVE) закриває `presence`-записи саме забраних, інші продовжують працювати з незмінним часом старту. Формальне "Завершити роботи" для обʼєкта закриває всіх, хто ще лишився відкритим.

---

## 8. Сценарій: Обʼєкти та маршрут

Порядок обʼєктів у списку — порядок вибору під час планування, він **довідковий**: DRIVE показує підказку "прямуємо до …" за цим порядком, але кожен рядок маршруту — самостійна точка входу, і бригадир може відкрити будь-який обʼєкт, у будь-який момент, скільки завгодно разів.

- **Скасували обʼєкт** — 🗑 прямо в списку на DRIVE (або в "Роботи"/"Обʼєкти" з HUB); якщо на ньому вже є роботи чи присутність — підтверджується через Undo-тост, не жорстке попередження.
- **Додали новий обʼєкт** — "➕ Додати обʼєкт" прямо на DRIVE, без виходу з екрана.
- **Повернулись на попередній обʼєкт** — тап по рядку в маршруті (DRIVE) чи в блоці "Маршрут" (HUB) відкриває "На обʼєкті" для нього незалежно від `visited`; "Почати роботи" відкриває нові `presence`-записи для тих, хто там зараз.
- **Частина людей працює, частина вже поїхала** — див. розділ 7; обʼєкт лишається "активним" (є відкриті записи), доки не закриті явно.

---

## 9. Сценарій: Робота на обʼєкті

Одна дія "▶️ Почати роботи" охоплює всіх, хто зараз висаджений (`here`), і всі заплановані для обʼєкта роботи одночасно — окремого вибору "яку саме роботу і хто саме" немає, бо в реальності бригада, висаджена на обʼєкті, працює над пакетом робіт разом. Завершення ("⏹ Завершити") закриває всі відкриті записи цього обʼєкта й одразу веде до введення обсягів по кожній роботі. Обсяги не обовʼязкові одразу — можна відкласти й дозаповнити пізніше (з обʼєкта повторно чи з екрана "Повернення", де є явна підказка про незаповнене).

---

## 10. Сценарій: Повернення і підсумок

На шляху до бази екран збору людей перевіряє, що на обʼєктах нікого не лишилось. Люди, які приїхали бусом, повертаються в бус групою. Серед тих, хто прибув сам, бригадир спочатку окремо садить у бус потрібних людей, а решту однією дією знімає з обʼєкта як таких, що їдуть самостійно. Обʼєкт із такими людьми показується в поверненні навіть тоді, коли основний бус не відвідав його на виїзному маршруті. Після розподілу всіх людей стає доступною дія "🏁 Приїхали на базу". Екран дозволяє дозаповнити обсяги, одометр фінішу валідується проти старту. REVIEW показує повний підсумок з реальними годинами (сума `presence`) і дозволяє скоригувати коефіцієнти перед відправкою. Відправка — з явним підтвердженням.

---

## 11. Розрахунок оплати праці

Незмінна формула: бригадир отримує 20% фонду кожного обʼєкта, де він працював (`presence` містить його запис); старші ділять 10%; робітники ділять решту пропорційно балам (`години × коефіцієнт дисципліни × коефіцієнт продуктивності`, коефіцієнти денні, застосовуються однаково на всіх обʼєктах). Доплата за виїзд — фіксована сума за клас поїздки (S/M/L/XL за пробігом одометра), порівну між працівниками зі складу дня, які не позначені у `selfTransportIds`. Якщо працівник приїхав сам, але повертається бусом, це не змінює його класифікацію для дорожньої доплати за цю поїздку.

Години кожного працівника на кожному обʼєкті = сума `(to ?? now) − from` по всіх його `presence`-записах саме на цьому обʼєкті — коректно відображає часткові виїзди й повторні заходи.

---

## 12. Статус звіту

Три реальні стани звіту: **не відправлено**, **відправлено (SUBMITTED)**, **затверджено (APPROVED)**. Лише `APPROVED` — read-only. `SUBMITTED` лишається повністю видимим і редагованим: повторне відкриття табеля того ж дня підвантажує останню відправлену версію в редаговану форму (якщо немає свіжішого локального чернетки), повторна відправка перезаписує ті самі рядки дня й додає новий запис у журнал подій для аудиту. Перехід у `APPROVED` — зовнішня дія адміністратора; мінідодаток лише читає цей статус. Для вже затвердженого дня — єдина дія "Запросити редагування", яка сповіщає адміністратора, не розблоковує автоматично.

---

## 13. Автозбереження, офлайн, аварійне відновлення

Весь `DayState` серіалізується в локальне сховище пристрою після кожної значущої зміни (дебаунс не потрібен — обсяг даних малий). При відкритті табеля: якщо є свіжий локальний чернетка на сьогодні — відновлюється він (найновіші, ще не відправлені зміни мають пріоритет); інакше, якщо на сервері є невідправлена версія — підвантажується вона; інакше — чистий день.

Це свідомий компроміс, не недогляд: пріоритет локального чернетки означає, що коли той самий бригадир того самого дня працює з **двох різних пристроїв**, новіші зміни з другого пристрою не побачать чернетку першого, доки той не буде явно відправлений. Для типового сценарію (один бригадир — один телефон увесь день) це не проблема; узгодження між пристроями посеред дня — поза скоупом.

**Довідники (авто/люди/обʼєкти/роботи) кешуються локально** після першого успішного завантаження й використовуються як резерв, якщо свіжий запит не вдався — застосунок лишається функціональним навіть при відкритті без інтернету взагалі (не лише посеред уже розпочатого дня).

Мережеві POST-запити повторюються з відкладенням при суто мережевих збоях (не при змістовних помилках сервера — ті показуються одразу). Індикатор стану синхронізації видно лише коли він відрізняється від "усе гаразд" (не займає постійно місце на екрані).

Аварійне закриття Telegram (вивантаження застосунку з памʼяті) не призводить до втрати дня — при повторному відкритті день відновлюється автоматично з того ж пристрою.

---

## 14. Захист від помилок

- Одометр фінішу не може бути меншим за старт — перевіряється перед переходом до підсумку, з поясненням.
- Фінальна відправка — тільки після явного підтвердження (модальне вікно Telegram, не звичайний тап).
- Видалення обʼєкта чи людини, по яких уже є дані, — з можливістю скасувати одразу (Undo-тост), а не мовчки.
- Зайнятість авто/людей іншим бригадиром — видно одразу, оновлюється періодично, а не лише в момент першого відкриття екрана.

## 15. Telegram Mini App інтеграція

- **BackButton** — нативна апаратна/жестова кнопка "назад" відображає той самий перехід, що й кнопка в застосунку; ніколи не залишає непередбачувано, не втрачаючи стан.
- **HapticFeedback** — тактильний відгук на вибір/підтвердження/помилку.
- **MainButton** — нативна кнопка Telegram використовується для екранів з **одним** однозначним головним дійством (виїхати, прибути на базу, відправити звіт) — зокрема відправка звіту користується вбудованим `showProgress()` замість ручної підміни тексту. Екрани зі складеною нижньою частиною (підказка + кнопка, кілька контекстних дій одночасно) лишаються на власному, кастомному компоненті кнопки — нативна кнопка Telegram глобальна й одна, тому не підходить для складених станів.
- **Popup API** (`showConfirm`) — для рішень з реальними наслідками: підтвердження фінальної відправки, підтвердження "почати день заново". Звичайні статусні повідомлення й помилки лишаються ненав'язливими інлайн-банерами, не модальними — Popup зарезервований саме для рішень, а не для щохвилинних статусів.
- **Viewport / Safe Area** — застосунок розгортається на весь viewport (`expand()`), нижня панель дій враховує safe-area на пристроях з жестовою навігацією.
- **Theme** — кольори йдуть з `--tg-theme-*` змінних, застосунок автоматично відповідає темній/світлій темі клієнта.

## 16. Масштабованість і продуктивність

- Групування людей і обʼєктів спирається на нормалізовані довідникові поля (`Brigade.name`, `WorkObject.city`), не на текстовий парсинг — коректно працює при десятках бригад і сотнях обʼєктів.
- Статус зайнятості авто/людей оновлюється періодично, поки бригадир на відповідному екрані вибору — знижує ризик колізій при одночасній роботі багатьох бригадирів.
- Фото (одометр, обʼєкти) стискаються на клієнті перед завантаженням (зменшення довшої сторони, помірна якість JPEG) — важливо саме в полі, на слабкому мобільному звʼязку.
- Списки довідників (сотні позицій) рендеряться без додаткової віртуалізації — обсяг даних цього типу для однієї компанії лишається малим для сучасного мобільного рендерингу; віртуалізація не потрібна, поки одиничний список не перевищує кількох тисяч позицій.

## 17. Серверний контракт

Мінімальний набір маршрутів, який реалізує описану поведінку. Google Таблиці лишаються джерелом істини для довідників і фактів дня; Postgres — дзеркало для читання; мінідодаток пише в обидва в межах одного запиту.

**Довідники (тільки читання):** авто, працівники (з `brigadeId`), бригади (`id → name`), роботи, обʼєкти (з `city`).

**Стан і блокування дня:**
- зайнятість авто на дату; зайнятість людей на дату — обидва опитуються повторно, поки бригадир на відповідному екрані вибору;
- раннє резервування авто/людей одразу після вибору (до завершення планування) — запобігає подвійному бронюванню кількома бригадирами одночасно;
- статус дня: чи є відправка, чи вона затверджена, чи є запит на редагування;
- останній відправлений день (для відновлення в редаговану форму);
- останній відправлений день **до вказаної дати** (для "повторити вчорашній маршрут");
- останній відомий одометр по кожному авто (для автопідстановки).

**Запис дня:**
- фінальне збереження (перезаписує рядки дня за ключем дата+бригадир+обʼєкт, додає новий запис у журнал подій — журнал не перезаписується, лишається аудитом);
- прев'ю розрахунку фонду без запису (для екрана підсумку до відправки);
- запит на редагування вже затвердженого дня (лише журналює подію для адміністратора);
- завантаження фото (одометр, обʼєкти) — приймає вже стиснене на клієнті зображення.

Усі POST-маршрути ідемпотентні відносно ключа (дата, бригадир, обʼєкт) — повторний виклик з тими самими даними не створює дублікатів у фактичних таблицях, лише в журналі подій (навмисно, для аудиту версій).

## 18. Межі системи (поза скоупом цієї специфікації)

Свідомо не входить у поточний продукт, але враховано архітектурно (не буде вимагати переписування, коли знадобиться):
- **GPS-автовизначення прибуття** — природно лягає на вже "вільний" вхід у будь-який обʼєкт (розділ 5, DRIVE/HUB): майбутній тригер просто викликає ту саму дію відкриття конкретного обʼєкта, яку зараз викликає тап.
- **Шаблони маршрутів** — розширення вже наявного механізму "повторити вчорашній маршрут" на довільну точку відліку, не лише "вчора".
- **PDF-звіти й аналітика** — вже мають готову структуровану основу (підсумок дня, журнал подій) для генерації пізніше без зміни моделі даних.
- **Голосове введення нотаток, AI-підказки** — додаткові способи введення в уже наявні поля (нотатки обʼєкта), не потребують нової моделі даних.

Не входить і не планується найближчим часом: QR-сканування, біометрія, інтеграції з зовнішніми системами — відсутність цих можливостей зараз не створює архітектурного боргу.
